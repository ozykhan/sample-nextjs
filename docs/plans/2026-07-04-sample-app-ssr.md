# sample-app — SSR Showcase Implementation Plan (Next.js)

> **For agentic workers:** Implement this plan task-by-task, test-first (failing test →
> implementation → green), one commit per step. Steps use checkbox (`- [ ]`) syntax for
> tracking.

**Goal:** Ship a tiny single-binary-ish Node web app that renders HTML server side with Next.js
(App Router + RSC), listens on one port (`$PORT`, default `8080`), exposes `/health`, and deploys
cleanly through Piper (`piper create sample-app && piper deploy sample-app --path .`) to serve at
`http://sample-app.piper.localhost`.

**Why:** Piper needs a readable showcase payload that exercises a real `Dockerfile`, the
`$PORT`-injection contract, the TCP health-check (accepting a connection), and Caddy's in-front
TLS — and a Next.js payload is the most realistic stack a contributor will *recognize*, since
"everyone uses Next.js these days." The Go version of this plan (rewritten here) was technically
fine but not representative of what people actually deploy. This keeps the Piper contract
identical (one container, one port, TCP health, Caddy-owned TLS) while using a mainstream stack
that is **self-hostable**, not Vercel-coupled — we use Next's `standalone` output and a plain
Node server, no Vercel-only features.

**Architecture:** One Next.js 15 app, App Router, TypeScript. `output: 'standalone'` in
`next.config.ts` produces a minimal self-contained `server.js` that reads `PORT` + `HOSTNAME`
env. Multi-stage `Dockerfile` (`node:22-alpine`) → runner image exposing `8080`. No Vercel
imagery, no ISR-on-edge, no image-optimization service — just a Node process on a port behind
Caddy, exactly the shape Piper expects.

**Tech Stack:** Node.js 22 LTS · Next.js 15 (App Router, RSC) · TypeScript · Vitest ·
`docker buildx` (multi-arch incl. arm64 for the Pi).

## Global Constraints

- **Runtime:** Node.js 22 LTS (`.nvmrc`). Builds run on Node 22.
- **Package manager:** npm (zero extra setup in `node:22-alpine`; pnpm/yarn are swappable but
  not used here to keep the Dockerfile corepack-free).
- **One port, `$PORT`.** Default `8080`. Next's standalone `server.js` honors `PORT` and
  `HOSTNAME` env directly; set `HOSTNAME=0.0.0.0` so the process binds all interfaces inside the
  container (Piper's Caddy proxies to the container's published host port).
- **Routes (exact):** `GET /` (home), `GET /about`, `GET /health`.
- **`/health` contract:** returns `200 OK`, `Content-Type: text/plain; charset=utf-8`, body
  `ok\n`, no external dependencies, no DB, no fetches. Must be reachable immediately on startup
  (Piper's TCP health-check dials the port; `/health` is the human-visible proof).
- **Commits:** one per step; conventional-commit style (`feat:`, `test:`, `chore:`).
- **No Piper import.** This app is a payload, not a client of Piper. No piper SDK, no piper CLI
  calls from the app.
- **No Vercel lock-in.** Forbidden: `@vercel/*` adapters, `edge` runtime routes,
  `next/image`'s default remote optimization loader, rewrites that assume edge middleware
  semantics. `next start` and the standalone `server.js` must run identically on any host.
- **No TLS in app.** Caddy (managed by Piper) terminates TLS in front. Never add
  redirect-to-HTTPS, HSTS headers, or `:443` listeners — that is Piper's job.

### Task 2: Home page (SSR)

**Files:** create `lib/version.ts`, `app/layout.tsx`, `app/page.tsx`, `app/globals.css`,
`tests/integration.test.ts`.

**Interfaces:**
- `lib/version.ts` exports `VERSION` (sourced from `package.json`, single source of truth for
  the footer).
- `app/layout.tsx` is the root layout: `<html>/<body>`, nav, `<main>{children}</main>`,
  `<footer>sample-app v{VERSION}</footer>`.
- `app/page.tsx` renders `Hello from sample-app` plus a line crediting Next.js SSR + Piper.

> **Why integration tests for pages (not RTL):** App Router pages are async RSCs; rendering
> them in jsdom via `react-dom/server` works but is fiddly and lies about `next/link`/image
> client boundaries. The realistic, low-maintenance gate is to build, start the standalone
> server on a random port, and `fetch` — exactly what a deployer sees. `Task 1`'s unit test
> covers the health handler directly; this task adds the end-to-end HTTP shape.

- [ ] **Step 1: Write the failing test** — `tests/integration.test.ts`:

```ts
import { describe, it, expect, beforeAll, afterAll } from 'vitest'
import { spawn } from 'node:child_process'
import { existsSync } from 'node:fs'

const SERVER = '.next/standalone/server.js'
const RUN = existsSync(SERVER)
const PORT = RUN ? 4321 + Math.floor(Math.random() * 1000) : 0
const base = `http://localhost:${PORT}`
let proc: ReturnType<typeof spawn> | undefined

async function waitFor(port: number, timeoutMs = 10000) {
  const deadline = Date.now() + timeoutMs
  while (Date.now() < deadline) {
    try {
      const r = await fetch(`http://localhost:${port}/health`)
      if (r.ok) return
    } catch { /* not up yet */ }
    await new Promise((r) => setTimeout(r, 100))
  }
  throw new Error(`server on ${port} did not become healthy`)
}

beforeAll(async () => {
  if (!RUN) return
  proc = spawn('node', [SERVER], {
    env: { ...process.env, PORT: String(PORT), HOSTNAME: '0.0.0.0' },
    stdio: 'ignore',
  })
  await waitFor(PORT)
}, 30000)

afterAll(() => {
  if (proc) proc.kill('SIGTERM')
})

describe.runIf(RUN)('integration against standalone server', () => {
  it('home renders SSR with title + version footer', async () => {
    const r = await fetch(`${base}/`)
    expect(r.status).toBe(200)
    const html = await r.text()
    expect(html).toContain('Hello from sample-app')
    expect(html).toContain('sample-app v')
  })
})
```
If `.next/standalone/server.js` doesn't exist (no build yet) the suite is *skipped*, not
failed — so `npm test` is fast and safe to run pre-build. CI builds before testing (Task 6)
so the integration suite actually executes there.

### Task 5: `Dockerfile` (multi-stage, standalone, no Vercel)

**Files:** `Dockerfile`, `.dockerignore`.

- [ ] **Step 1: Write the Dockerfile**

```dockerfile
# deps
FROM node:22-alpine AS deps
WORKDIR /app
COPY package.json package-lock.json* ./
RUN npm ci

# builder
FROM node:22-alpine AS builder
WORKDIR /app
COPY --from=deps /app/node_modules ./node_modules
COPY . .
ENV NEXT_TELEMETRY_DISABLED=1
RUN npm run build

# runner
FROM node:22-alpine AS runner
WORKDIR /app
ENV NODE_ENV=production
ENV NEXT_TELEMETRY_DISABLED=1
ENV HOSTNAME=0.0.0.0
ENV PORT=8080
RUN addgroup -g 1001 -S nodejs && adduser -S nextjs -u 1001
COPY --from=builder --chown=nextjs:nodejs /app/.next/standalone ./
COPY --from=builder --chown=nextjs:nodejs /app/.next/static ./.next/static
COPY --from=builder --chown=nextjs:nodejs /app/public ./public
USER nextjs
EXPOSE 8080
ENTRYPOINT ["node", "server.js"]
```
Notes:
- Standalone's `server.js` reads `PORT` and `HOSTNAME` env; `HOSTNAME=0.0.0.0` is mandatory so
  the listener is reachable from the host port Piper publishes (Next defaults to `localhost`
  inside a container, which would make Piper's TCP dial fail).
- `ENV PORT=8080` is the contract default; Piper overrides it with the configured app port at
  `docker run` time, which is fine — env at runtime beats the Dockerfile default.
- No `next/image` remote loader, no edge runtime, no Vercel adapters — runs anywhere Node runs.

### Task 6: CI `verify` workflow + `Makefile`

Mirror Piper's gate (lint → typecheck → build → test) so `main` stays green/releasable, plus
an arm64 buildx cross-arch check standing in for the old Go `make cross` (Pi target sanity).

**Files:** `.github/workflows/verify.yml`, `Makefile`.

- [ ] **Step 1: Workflow**

```yaml
name: verify
on:
  push:
    branches: [main]
  pull_request:
    branches: [main]
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with:
          node-version: '22'
          cache: 'npm'
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck
      - run: npm run build
      - run: npm test          # integration suite now runs (build artifacts present)
  cross:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: docker/setup-buildx-action@v3
      - run: docker buildx build --platform linux/arm64 .
```
Note build is ordered before test so the integration suite (which needs
`.next/standalone/server.js`) actually runs in CI. The `cross` job validates the Dockerfile
builds for arm64 (Pi parity) without loading the image locally.

- [ ] **Step 2: `Makefile`** (convenience + parity with Piper's `make` muscle memory):
```makefile
.PHONY: dev build test run cross
dev:
	npm run dev
build:
	npm run build
test:
	npm test
run:
	npm run start
cross:
	docker buildx build --platform linux/arm64 .
```

- [ ] **Step 3: Commit**
